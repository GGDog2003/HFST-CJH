# IXI 低 PSNR 问题分析与改进说明

## 1. 现象

- 你当前跑到 `14000 epoch` 时，IXI test 的 `PSNR` 约为 `31.23 dB`。
- 作者原版模型文件在同类测试上可以到 `32.25 dB` 左右。
- 这说明当前实现和作者可复现实验之间，至少还存在一层“数据链路差异”或“网络实现差异”。

## 2. 我认为最可能的原因

### 2.1 数据归一化破坏了跨模态强度关系

当前 `DatasetIXIT` 里有两个会一起拉低泛化的点：

1. `T1` 和 `T2` 先走的是联合 `min-max` 归一化。  
这会让某一个模态的极值直接影响另一个模态的缩放比例。

2. 退化后的 `LR T2` 又在 `degrade()` 里单独再做了一次归一化。  
这会让输入 `LR T2`、参考 `HR T1`、目标 `HR T2` 三者之间的强度对应关系被再次打散。

对于这种“`LR T2 + HR T1 -> HR T2`”的跨模态超分任务，这种强度关系不稳定会比普通单模态超分更伤。

### 2.2 当前主干已经不是“最稳的作者原版链路”

仓库里现在同时存在：

- `models/network_release.py`
- `models/network_release_bak.py`
- `models/network_release_refine.py`

这通常说明网络结构已经经历过几轮实验性改动。  
如果你当前训练时走的是被多次叠改后的版本，而作者 `32.25 dB` 的结果来自更稳的实现，那么出现 `1 dB` 左右差距是合理的。

### 2.3 HH 早期子域很可能存在明显域偏移

你前面那份分析已经很清楚：  
`HH-1211 ~ HH-1359` 这一批样本明显掉点，而 `HH-1400+` 又恢复正常。

这更像是：

- 采集协议变化
- 强度分布变化
- 跨模态对应关系变化

而不是“模型整体不会做 HH”。

### 2.4 评估链路里存在会误导判断的实现风险

我排查时还看到两个额外隐患：

- `main_test_spsr.py` 原来不能优雅加载“新结构 + 旧 checkpoint”。
- `models/model_plain.py` 的 tile 推理分支对多输入模型不安全，而且 patch 回填代码本身有问题。

它们不一定是你这次 `31.23 dB` 的直接根因，但会让后续实验判断变得不稳定。

## 3. 这次已经落地的改动

## 3.1 数据层：把 IXI 归一化改成可配置的稳健方案

修改文件：

- `data/dataset_sr.py`
- `options/train/x4/train_ixit_spsr_release_s4_d96_w5_n3.json`
- `options/test/x4/test_ixit_spsr_release_s4_d96_w5_n3.json`

具体改动：

1. 给 `DatasetIXIT` 增加了三种可切换归一化模式：

- `joint_minmax`
- `independent_minmax`
- `independent_percentile`

2. 当前训练/测试配置已经切到：

- `normalize_mode = independent_percentile`
- `normalize_percentiles = [0.5, 99.5]`
- `lr_renormalize = false`

3. 这样做的目的：

- 不再让 `T1/T2` 互相污染缩放比例
- 用分位数裁掉异常极值
- 让退化后的 `LR T2` 尽量保留和 `HR T2` 一致的强度尺度

## 3.2 网络层：切到“稳定主干 + 轻量细化头”

相关文件：

- `models/network_release_refine.py`
- `models/select_network.py`
- `models/model_base.py`

这条链路的核心思路是：

1. 主干直接复用更稳定的 baseline 主体。
2. 在输出端增加一个轻量 `ResidualReliabilityRefineHead`。
3. 细化头同时看：

- 基础重建结果
- 上采样后的 `LR`
- 参考模态 `Ref`
- 主分支梯度
- 参考模态梯度

4. 通过可靠性门控抑制“参考模态错误引导”的区域，只在高可信区域加残差修正。

这个方案相比继续在主干中部叠很多大模块，更适合你现在这个阶段：

- 可以继续吃旧 checkpoint
- 改动更收敛
- 更容易判断到底有没有真实收益

## 3.3 权重加载层：允许新结构复用旧 checkpoint

修改文件：

- `models/model_base.py`
- `main_test_spsr.py`

具体改动：

1. 训练加载阶段如果模型实现了 `load_pretrained()`，就走“部分参数映射加载”。
2. 测试入口也补了同样的逻辑。

这样你可以直接：

- 用旧的 `SPSRNet_release` checkpoint
- 去初始化新的 `SPSRNet_release_refine`

而不是强制从零开始训。

## 3.4 推理层：修掉 tile 分支的错误回填

修改文件：

- `models/model_plain.py`

具体改动：

1. 单输入模型时，patch 输出不再错误地写回自身缓存。
2. 多输入模型时，如果仍配置了 tile，会直接回退整图推理，避免错误拼接。

这条改动主要是为了防止后面排查时再被无关 bug 干扰。

## 4. 现在配置已经怎么切了

训练配置：

- `options/train/x4/train_ixit_spsr_release_s4_d96_w5_n3.json`

测试配置：

- `options/test/x4/test_ixit_spsr_release_s4_d96_w5_n3.json`

当前都已经切成：

- `net_type = SPSRNet_release_refine`
- `independent_percentile` 归一化
- `lr_renormalize = false`

也就是说，你后面继续训练/测试时，会直接走这套改进版链路。

## 5. 我对后续实验的建议

### 5.1 先不要直接拿旧的 31.23 和新配置做一对一结论

因为这次不只是改了网络，还改了数据归一化。  
新结果更像是“新训练方案”的起点，而不是“旧 checkpoint 立刻白嫖涨点”。

### 5.2 最好做两组最小消融

建议至少拆成：

1. 只改归一化，不改网络
2. 归一化 + refine head 一起改

这样你能很快看清：

- 是数据链路贡献更大
- 还是细化头贡献更大

### 5.3 HH 子域建议单独报表

建议把测试结果至少拆成：

- `Guys`
- `HH-1211~1359`
- `HH-1360+`

如果总均值涨了但 `HH-1211~1359` 还很差，那说明主问题仍然是域偏移，没有真正解决。

## 6. 仍然存在的风险

1. `network_release_refine.py` 现在是“旧 checkpoint 复用 + 新头继续训练”的策略。  
如果不继续训练，只做直接测试，随机初始化的细化头不会带来真实收益。

2. `independent_percentile` 是否优于 `independent_minmax`，还需要你用同一训练预算做消融确认。

3. 当前我没有替你完整重跑一遍训练和全量 IXI 测试，所以这次提交解决的是“最可疑的实现问题”和“更稳的下一版实验入口”，不是直接保证最终分数已经超过 `32.25 dB`。

## 7. 建议你下一步怎么跑

### 继续训练

```bash
python main_train_spsr.py --opt options/train/x4/train_ixit_spsr_release_s4_d96_w5_n3.json
```

### 测试

```bash
python main_test_spsr.py --opt options/test/x4/test_ixit_spsr_release_s4_d96_w5_n3.json -d cuda:0
```

如果你想做“只比归一化”的纯消融，我建议再复制一份配置，把：

- `net_type` 改回 `SPSRNet_release`

这样可以更快看出到底是不是数据链路在拖分。
